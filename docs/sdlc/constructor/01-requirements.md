# Requirements: System Design Constructor

## 1. Описание фичи

Интерактивный пошаговый конструктор, в котором пользователь проектирует реальную систему, выбирая технологические решения на каждом этапе. После прохождения — получает итоговую архитектуру, trade-off анализ и сравнение с эталонным решением.

## 2. Функциональные требования

### FR-1: Каталог сценариев
- Отображение списка доступных сценариев с названием, описанием и сложностью
- Фильтрация по сложности (middle / senior)
- Маршрут: `/constructor`

### FR-2: Пошаговый конструктор
- Каждый сценарий состоит из 5–7 шагов, повторяющих структуру SD-интервью:
  1. Requirements Clarification (уточнение требований, capacity estimation)
  2. API Design (протокол, эндпоинты, sync/async)
  3. Data Model & Storage (БД, схема, партиционирование)
  4. High-Level Architecture (компоненты, взаимодействие)
  5. Scaling & Performance (кеширование, CDN, шардинг, репликация)
  6. Reliability & Monitoring (failover, circuit breaker, observability)
- На каждом шаге — набор решений (decisions) с вариантами выбора (options)
- Каждый вариант содержит: описание, pros, cons, когда лучше применять
- Некоторые решения допускают множественный выбор (multiSelect)
- Маршрут: `/constructor/[scenarioId]/session`

### FR-3: Живая сводка метрик
- Панель с метриками системы: latency, scalability, consistency, complexity, cost
- Метрики пересчитываются при каждом выборе пользователя на основе impact выбранных опций
- Визуализация: radar chart (SVG) или таблица с индикаторами

### FR-4: Подсказки
- Каждый шаг может содержать подсказку (tip)
- Подсказка скрыта по умолчанию, раскрывается по клику

### FR-5: Навигация по шагам
- Прогресс-бар сверху с отображением текущего шага
- Возможность вернуться к любому пройденному шагу и изменить выбор
- При изменении выбора на предыдущем шаге — метрики пересчитываются

### FR-6: Итоговый экран
- Собранная архитектура на основе выборов пользователя (текстовое описание + ASCII-диаграмма)
- Trade-off матрица: что получили vs чем пожертвовали
- Сравнение с эталонным решением: подсветка совпадений и расхождений
- Детальные пояснения по каждому решению (почему эталон выбрал иначе)
- Общая оценка (score) на основе близости к эталону
- Кнопка «Пройти заново» — сброс и возврат к шагу 1

### FR-7: Сохранение прогресса
- Сохранение пройденных сценариев и выборов в localStorage
- На странице каталога — отметка пройденных сценариев и последний score

### FR-8: Сценарии

8 сценариев для первой версии:

|----|---------|-----------|

## 3. Нефункциональные требования

### NFR-1: Без AI на runtime
- Все варианты, оценки и эталонные решения предрассчитаны в данных
- Мгновенный отклик, работа без API-ключа

### NFR-2: Консистентность с существующим UI
- Используется та же дизайн-система: Tailwind, тёмная тема, Header, ProgressBar
- Общий стиль карточек и типографики

### NFR-3: Адаптивность
- Корректное отображение на мобильных устройствах
- На десктопе: двухколоночный layout (шаг + сводка метрик)
- На мобильных: одноколоночный, сводка метрик сворачивается

### NFR-4: Производительность
- Данные сценариев загружаются статически (import), без API-вызовов
- Переключение шагов без перезагрузки страницы

### NFR-5: Расширяемость
- Структура данных позволяет добавлять новые сценарии без изменения кода компонентов
- Каждый сценарий — отдельный файл в `src/data/constructor/`

### NFR-6: Тестируемость
- Компоненты и хуки покрыты unit-тестами
- Данные сценариев валидируются тестами на соответствие типам

### NFR-7: Достоверность контента
- Все архитектурные решения, trade-offs, pros/cons и эталонные решения должны соответствовать реальным production-практикам
- Источники: публичные engineering-блоги (Meta, Google, Uber, Netflix, Cloudflare, Stripe, Airbnb, LinkedIn, Yandex)
- Запрещены выдуманные технологии, несуществующие паттерны, упрощения искажающие суть
- Impact-метрики должны отражать реальные trade-offs (CAP-теорема, реальные latency характеристики)
- Каждое эталонное решение — то, что реально используется в production при заданном масштабе

## 4. Ограничения

- Только клиентский рендеринг для интерактивных компонентов (Client Components)
- localStorage как единственное хранилище прогресса (без бэкенда)
- Первая версия без AI-оценки открытых комментариев пользователя к своим выборам

## 5. Вне скоупа (v1)

- AI-генерация сценариев
- Экспорт результатов (PDF, share link)
- Пользовательские сценарии (custom)
- Диаграммы в формате Mermaid/Excalidraw (только ASCII/текст)
- Таймер на прохождение
